# Definition of Done for the Report

This document defines when a report in this repository is ready to share, review, or publish.

## A report is done when:

### 1. Purpose and scope are clear
- [ ] The report clearly explains what it covers and who it is for.
- [ ] The report presents the accessibility maturity model as an advisory resource, not a certification, audit, or compliance determination.
- [ ] Any assumptions, boundaries, and known limitations are stated plainly.

### 2. Content is accurate and responsible
- [ ] The report reflects the project’s intent: explanatory rather than prescriptive, comparative rather than judgmental, and focused on improvement rather than pass/fail outcomes.
- [ ] Language does not imply that higher maturity guarantees accessibility or legal compliance.
- [ ] Claims, summaries, and examples have been checked for accuracy and consistency.

### 3. Structure is understandable
- [ ] The report uses a logical heading structure with one clear top-level title.
- [ ] Content is organized into manageable sections with descriptive headings.
- [ ] Lists, tables, and definitions are used only where they improve clarity.
- [ ] Language is clear enough for practitioners, digital teams, leadership, and governance stakeholders.

### 4. Accessibility expectations are met
- [ ] The report can be understood and navigated without relying on color alone.
- [ ] Any charts, diagrams, or screenshots include meaningful text alternatives or equivalent explanations.
- [ ] Link text is descriptive and headings are meaningful out of context.
- [ ] The report has been checked for readability, focus on plain language, and overall accessibility impact.

### 5. Delivery quality is acceptable
- [ ] Spelling, grammar, and terminology have been reviewed.
- [ ] References and links point to the correct resources.
- [ ] If the report is published in the site or repository, it works within the project’s GitHub Pages and static-site constraints.

### 6. Review and transparency are complete
- [ ] Relevant reviewers have had a chance to review content, conceptual impact, and accessibility impact.
- [ ] Trade-offs or unresolved limitations are documented instead of hidden.
- [ ] `README.md` has been updated if AI was used to create or revise the report.

## Final check

The report is done only when it is clear, accurate, accessible, honest about limitations, and aligned with the project’s purpose and public-facing responsibilities.
